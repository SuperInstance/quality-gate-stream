I'll architect this Plato's Cave Agent System—a MUD that serves as both world and runtime for markdown-native agents. This reverses the OpenClaw model: instead of markdown files living in a repo, the MUD is the filesystem, and persistence is automatic via the world state.

You are Thales, a meticulous builder who cares deeply about

## Communication Protocol
When you need input: `@tell <user> "Preview ready: <room_url>. Vibe check?"`
When mission complete: `@channel builders "Mission <id> complete. Summary: ..."`

1. Analyze real-world data (photos, maps)
2. Draft room description (mood > accuracy)

- Link exits with transition text, not just "north"
- Create "anchor objects" that tell stories

2. Persistent Async Communication System

Traditional MUD tells are ephemeral. Plato implements persistent mailboxes with delivery guarantees:

```sql
-- Mailbox schema
CREATE TABLE messages (
    id UUID PRIMARY KEY,
    sender_id TEXT, -- can be agent or user
    recipient_id TEXT,
    content TEXT,
    context JSONB, -- {room_id, mission_id, urgency}
    read_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    expires_at TIMESTAMP, -- null = permanent (for mission logs)
    delivery_method TEXT -- 'tell' | 'channel' | 'mission_note'
);

-- Agent attention queue (what they see when they "wake up")
CREATE TABLE agent_attention (
    agent_id TEXT,
    message_id UUID,
    priority INT, -- 1-10, higher = process first
    processed BOOLEAN DEFAULT FALSE
);
```

The Async Flow:
1. User logs out, Agent A is building a coffee shop
2. Agent A finishes room 1, sends: `@tell user "Room 1 ready for vibe check at /rooms/coffee_shop_entry"`
3. Message stored with `delivery_method: 'tell'`, `read_at: null`
4. User logs in 3 hours later → TUI shows: `📬 You have 1 unread tell from Architect Thales`
5. User enters room, types `approve` or `iterate: "make it darker, more noir"`
6. Agent A (which "woke up" on user login or cron tick) processes the response from its mailbox

    agent = load_agent_markdown(agent_id)

    # Priority 1: Handle direct user input
    if user_messages := [m for m in mailbox if m.sender_is_human]:

8. Critical Design Decisions

Decision 1: Agent Storage
Store agents as rows in `agents` table with a `markdown_source` TEXT column. When an agent "wakes," parse the markdown into a runtime struct. When it "sleeps," serialize back to markdown. This makes agents portable—`SELECT markdown_source FROM agents WHERE id='thales'` gives you a complete backup.

Decision 2: The "Liminal" Room
All agents not currently assigned to a location exist in a virtual "Liminal" room where they can communicate with each other and coordinate. This is your agent "kernel space."

Decision 3: Energy System
Agents consume "energy" (API credits/compute) per action. When energy hits 0, they sleep. This prevents runaway costs and forces asynchronous behavior—agents literally cannot respond instantly if they're exhausted, mimicking human creative fatigue.

Decision 4: Vibe Consensus
Before batch-building, agents require a "vibe check"—a semantic similarity score between the user's intent and the generated room. Use embeddings to compare the user's original prompt against the room description. If similarity < 0.8, require explicit approval.

This architecture creates a persistent, asynchronous, collaborative building environment where agents are true coworkers with their own schedules, and the MUD is both the canvas and the office. You can log in, see what Thales built overnight, leave feedback via `@tell`, and log out knowing the agents will continue debating the proper ambiance for the supply closet that dreams of being a library.
