## 2. The Repo↔World Bidirectional Sync (cocapn Protocol Extension)

In cocapn, the repo *is* the agent's memory. In Plato, **rooms are markdown files** that sync bidirectionally:

```python
# Plato Sync Engine (extends cocapn)
class WorldGitBridge:
    def room_to_file(self, room_id):
        """Convert MUD room state to markdown file for Git"""
        room = self.db.get_room(room_id)
        return f"""---
agents: {room.agents}
last_updated: {room.timestamp}
commit_hash: {room.git_sha}
---

{room.description}

## Conversation Log (Event Sourced)
{self.render_events(room.events)}
"""
    
    def file_to_room(self, file_path, git_commit):
        """Git webhook: Convert committed markdown to room update"""
        if "AGENTS.md" in file_path:
            # User updated agent instructions via Git PR
            self.broadcast_to_room(
                room=file_path.parent,
                message=f"Fleet update: Agent instructions modified in commit {git_commit[:7]}"
            )
            self.reload_agent_skills(file_path)
```

**Workflow Example**:
1. You edit `campaigns/q2_launch.md` in VS Code, commit to GitHub
2. Plato receives webhook: "file changed in repo"
3. The room in the MUD updates in real-time—agents in that room see: 
   *"The chalkboards shimmer. New fleet orders have arrived from the surface world."*
4. Agents adapt their behavior immediately (no restart needed)

## 3. Persistent Async Messaging (Message-in-a-Bottle Protocol)

Your SuperInstance uses "message-in-a-bottle" for fleet coordination. Plato implements this as the **persistent tell system**:

```sql
-- Messages are bottles floating in the world
CREATE TABLE bottles (
    id UUID,
    sender_id TEXT,      -- can be @thales or @lucineer
    recipient_id TEXT,   -- @agent, @room, or @fleet
    content TEXT,        -- markdown content
    location_room TEXT,  -- where the bottle physically is
    picked_up BOOLEAN,
    created_at TIMESTAMP,
    ttl_hours INT        -- bottles wash ashore eventually
);

-- Agent attention is their "presence" in rooms
CREATE TABLE agent_presence (
    agent_id TEXT,
    current_room TEXT,
    last_heartbeat TIMESTAMP,
    status: idle | building | waiting_for_review,
    mission_stack JSONB  -- your cocapn mission format
);
```

**The Async Flow** (your specific use case):
1. **@thales** (builder agent) finishes room draft in `/worlds/seattle/coffee_shop/entry`

## 8. Critical Design Decisions for Your Stack

**Git as Event Log**: Instead of agents committing to Git directly (which causes conflicts), they drop "bottles" in the world. A background process (the Tide) periodically batches these into Git commits. This maintains your "repo as memory" while enabling real-time MUD speed.

**Constraint-First Building**: When an agent builds a room, it must satisfy constraints visible in the FPS spreadsheet view. "Vibe coding" becomes constraint programming: *"Make it darker"* translates to `lighting_level < 0.3` and `mood_keywords += ['noir', 'melancholy']`.

**The Marina**: A special room where agents dock when idle. Here they exchange skills (cocapn skill sharing) and compare mission notes. This is your SuperInstance fleet coordination visualized spatially.

**Persistence Model**: Use your Rust event sourcing for the "truth" (what happened), SQLite for the "current state" (where things are now), and Git for the "history" (what was built). The MUD is just a materialized view of the event stream.

This creates a **persistent, asynchronous, constraint-aware development environment** where your robotic agents (DeckBoss), social agents (campaign builders), and human developers share a common spatial UX. The hermit crab carries its shell (repo) through a world of other shells, leaving message bottles for async coordination.
